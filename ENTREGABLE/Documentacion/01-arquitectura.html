3. Patrones de Diseño Aplicados

BFF (Backend For Frontend)

Nginx actúa como API Gateway unificado. El frontend hace todas las peticiones a un único host (:8080) y Nginx enruta por prefijo: /api/orders→8081, /api/inventory→8082, /api/shipments→8084, /api/notifications→8085.

Saga (Orquestación)

Al confirmar un pedido (PUT /api/orders/:id/confirm), orders-service orquesta en secuencia: (1) descuenta inventario, (2) crea envío en shipping-service. Los errores en pasos intermedios se registran como warnings sin revertir los pasos anteriores.

Database-per-Service

Cada microservicio tiene su propia base de datos PostgreSQL (orders_db, inventory_db, shipping_db, notification_db). No hay acceso cruzado entre bases de datos.

Repository + Shared Module

El módulo Backend/shared/ centraliza: creación del pool de conexiones (db.js), configuración de seguridad (security.js), logger (logger.js), validadores (validate.js) y el helper interServiceFetch.

4. Flujo de Confirmación de Pedido (Saga)

1

Frontend envía PUT /api/orders/:id/confirm al BFF.

2

orders-service consulta el pedido en orders_db.

3

orders-service → inventory-service: POST /api/inventory/:sku/adjust?delta=-qty para descontar stock.

4

orders-service → shipping-service: POST /api/shipments para crear el envío con número de tracking.

5

shipping-service → notification-service: POST /api/notifications con el evento SHIPMENT_CREATED.

6

orders-service actualiza el status del pedido a EN_PREPARACION y responde al frontend.

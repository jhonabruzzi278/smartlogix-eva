SmartLogix — Persistencia de Datos

Evaluación Parcial N°3  |  Estrategia: PostgreSQL + Stored Procedures (plpgsql)

1. Estrategia de Persistencia

SmartLogix utiliza PostgreSQL 15 como motor de base de datos relacional. La capa de persistencia combina dos enfoques:

  • Acceso directo vía SQL parametrizado: queries con $1, $2, ... usando el driver pg de Node.js (protección contra SQL injection).
  • Stored Procedures (funciones plpgsql): lógica de negocio compleja delegada a la base de datos para operaciones atómicas y reportes con JOIN.

Patrón Database-per-Service: cada microservicio tiene su propia base de datos independiente. No existe acceso cruzado entre bases de datos. El pool de conexiones se crea en el módulo compartido Backend/shared/db.js usando pg.Pool.

3. Stored Procedures (Funciones plpgsql)

SmartLogix implementa 4 stored procedures en PostgreSQL, creadas automáticamente al iniciar cada microservicio mediante la función ensureProcedures(). Usan la extensión plpgsql de PostgreSQL.

3.1 fn_adjust_stock — inventory_db

Propósito: Ajustar el stock de un SKU de forma atómica. Retorna un resultado estructurado indicando éxito o tipo de error (SKU no encontrado vs. stock insuficiente).

CREATE OR REPLACE FUNCTION fn_adjust_stock(p_sku TEXT, p_delta INT)
RETURNS TABLE(
    sku_out TEXT, new_stock INT, delta INT,
    success BOOLEAN, error_msg TEXT
)
AS $fn$
DECLARE
    v_new_stock INT;
    v_exists   BOOLEAN;
BEGIN
    -- Verificar existencia del SKU
    SELECT EXISTS(SELECT 1 FROM inventory WHERE sku = p_sku) INTO v_exists;

    IF NOT v_exists THEN
        RETURN QUERY SELECT p_sku, NULL::INT, p_delta,
                            FALSE, 'SKU no encontrado'::TEXT;
        RETURN;
    END IF;

    -- Actualizar solo si stock + delta >= 0 (evita stock negativo)
    UPDATE inventory
    SET stock = stock + p_delta
    WHERE sku = p_sku AND stock + p_delta >= 0
    RETURNING stock INTO v_new_stock;

    IF v_new_stock IS NOT NULL THEN
        RETURN QUERY SELECT p_sku, v_new_stock, p_delta,
                            TRUE, NULL::TEXT;
    ELSE
        RETURN QUERY SELECT p_sku, NULL::INT, p_delta,
                            FALSE, 'Stock insuficiente'::TEXT;
    END IF;
END;
$fn$ LANGUAGE plpgsql;

Invocación desde Node.js:

const r = await pool.query('SELECT * FROM fn_adjust_stock($1,$2)', [sku, delta]);
const result = r.rows[0];
if (!result.success) {
    const status = result.error_msg === 'SKU no encontrado' ? 404 : 400;
    return res.status(status).json({ error: result.error_msg });
}

3.2 fn_get_inventory_report — inventory_db

Propósito: Clasificar todos los productos por nivel de stock en tiempo real.

CREATE OR REPLACE FUNCTION fn_get_inventory_report()
RETURNS TABLE(sku VARCHAR, stock INT, stock_level TEXT)
AS $fn$
BEGIN
    RETURN QUERY
        SELECT i.sku, i.stock,
               CASE
                   WHEN i.stock = 0    THEN 'SIN_STOCK'
                   WHEN i.stock < 10   THEN 'CRITICO'
                   WHEN i.stock < 30   THEN 'BAJO'
                   ELSE                      'NORMAL'
               END::TEXT
        FROM inventory i
        ORDER BY i.stock ASC;
END;
$fn$ LANGUAGE plpgsql;

3.3 fn_get_orders_with_customer — orders_db

Propósito: Obtener órdenes con datos del cliente (JOIN entre orders y customers), con filtro opcional por status.

CREATE OR REPLACE FUNCTION fn_get_orders_with_customer(p_status TEXT DEFAULT NULL)
RETURNS TABLE(
    order_id INT, customer_name VARCHAR, customer_email VARCHAR,
    sku VARCHAR, quantity INT, status VARCHAR,
    created_at TIMESTAMP, assigned_to VARCHAR
)
AS $fn$
BEGIN
    RETURN QUERY
        SELECT
            o.id,
            COALESCE(c.name, 'Sin cliente'),
            COALESCE(c.email, ''),
            o.sku, o.quantity, o.status,
            o.created_at, o.assigned_to
        FROM orders o
        LEFT JOIN customers c ON c.id = o.customer_id
        WHERE p_status IS NULL OR o.status = p_status
        ORDER BY o.created_at DESC;
END;
$fn$ LANGUAGE plpgsql;

3.4 fn_cancel_order — orders_db

Propósito: Cancelar una orden de forma atómica: UPDATE + SELECT en una sola transacción usando SETOF. Retorna la orden actualizada.

CREATE OR REPLACE FUNCTION fn_cancel_order(
    p_order_id INT, p_reason TEXT DEFAULT ''
)
RETURNS SETOF orders
AS $fn$
BEGIN
    -- Actualizar a CANCELADO (solo si no está ya cancelada)
    UPDATE orders
    SET status = 'CANCELADO', cancel_reason = p_reason
    WHERE id = p_order_id AND status <> 'CANCELADO';

    -- Retornar la fila actualizada
    RETURN QUERY SELECT * FROM orders WHERE id = p_order_id;
END;
$fn$ LANGUAGE plpgsql;

5. Pool de Conexiones (Backend/shared/db.js)

Todos los microservicios usan el módulo compartido para crear el pool PostgreSQL:

// Backend/shared/db.js
const { Pool } = require('pg');

function createPool(dbName) {
  return new Pool({
    host:     process.env.DB_HOST || 'localhost',
    port:     process.env.DB_PORT || 5432,
    database: process.env.DB_NAME || dbName,
    user:     process.env.DB_USER || 'postgres',
    password: process.env.DB_PASSWORD || 'postgres',
    max:      10,
    idleTimeoutMillis: 30000,
  });
}

module.exports = { createPool };

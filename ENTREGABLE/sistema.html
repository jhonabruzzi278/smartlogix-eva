1

¿Por qué microservicios para SmartLogix?

SmartLogix modela el ciclo completo de un pedido logístico: creación de orden → gestión de stock → despacho → entrega confirmada. Estas cuatro operaciones tienen ritmos, volúmenes y requisitos de disponibilidad radicalmente distintos. Una arquitectura monolítica las fundiría en un único proceso donde un pico de pedidos bloquearía al módulo de inventario, una falla en las notificaciones detendría los envíos y un deploy forzaría downtime total. Los microservicios resuelven exactamente eso.

Razón #1

Dominios aislados

Órdenes, inventario, envíos y notificaciones son dominios de negocio independientes con modelos de datos distintos. Cada servicio tiene su propia base de datos PostgreSQL y su propio esquema de tablas — no comparten estado.

Razón #2

Despliegue independiente

Un cambio en la lógica de shipping no requiere redesplegar órdenes ni inventario. En producción esto significa releases más frecuentes, rollbacks precisos y menor riesgo por cada deploy.

Razón #3

Escalado selectivo

Las consultas de tracking pico en temporadas altas (Black Friday). Con microservicios, solo se escala orders-service horizontalmente — los demás servicios no consumen recursos innecesarios.

Razón #4

Resiliencia por aislamiento

Si notification-service cae, los envíos siguen funcionando. El shipping-service registra el intento de notificación fallido pero completa la creación del envío — demostrado en los tests de resiliencia.

Razón #5

Tecnologías especializadas

Cada servicio puede evolucionar su stack independientemente. El módulo de notificaciones puede migrar a un broker de mensajes (RabbitMQ, SQS) sin tocar el código de órdenes.

Razón #6

Contextos delimitados claros

Aplicamos el principio de Bounded Contexts de DDD: la entidad "orden" en orders-service tiene un significado distinto al "pedido" que shipping-service convierte en "envío" con número de tracking propio.

Frente a un monolito: En un monolito, una transacción que toca órdenes + inventario + envíos dentro del mismo proceso puede tardar segundos y bloquear recursos. En SmartLogix, el Saga orquestado distribuye ese trabajo entre tres servicios independientes que operan con sus propias transacciones, garantizando consistencia eventual sin lock global.

Importancia de cada microservicio

:8081 orders-service — El orquestador central

Es el núcleo de SmartLogix. Todo pedido nace, vive y muere en este servicio. Su rol más crítico es actuar como orquestador del Saga: cuando una orden se confirma, este servicio coordina en secuencia a los otros dos (inventory y shipping), maneja sus respuestas y decide si la transacción avanza o emite advertencias.

Por qué es irremplazable

• Genera el código SL-XXXXXX de cliente — identificador único que el comprador usa para rastrear su pedido, validar la entrega y que el sistema usa como prueba de propiedad del paquete.

• Implementa RLS (Row-Level Security) server-side: extrae el rol del JWT y elimina client_code del response para roles restringidos (shipper, customer, vendor), garantizando que el transportista nunca acceda al código privado del cliente.

• Expone el endpoint público de tracking /api/orders/track/:clientCode que devuelve solo campos seguros — sin email ni datos de contacto — permitiendo rastreo sin autenticación.

• Gestiona la lógica de compensación: al cancelar una orden en preparación, llama al inventory-service para restaurar stock y al shipping-service para cancelar el envío activo.

Stored procedures que usa

fn_get_orders_with_customer — reporte de órdenes con JOIN a clientes  |  fn_cancel_order — cancela con motivo y timestamp

:8082 inventory-service — El guardián del stock

Controla la disponibilidad real de productos. Sin este servicio, podrían generarse órdenes para productos agotados, generando promesas imposibles de cumplir y devoluciones costosas.

Por qué es irremplazable

• Toda operación de ajuste de stock pasa por fn_adjust_stock, un stored procedure que valida atómicamente que el resultado no sea negativo — el stock no puede quedar en -1 aunque dos peticiones lleguen simultáneamente.

• El reporte fn_inventory_report clasifica cada SKU en SIN_STOCK / CRÍTICO / NORMAL, alertando al equipo de operaciones antes de que los problemas afecten al cliente.

• El módulo de POS (Punto de Venta) del frontend descuenta stock directamente vía /api/sales sin crear orden — un flujo separado para ventas presenciales que también valida stock disponible antes de registrar la venta.

• Es el receptor de las llamadas de compensación del Saga: cuando una orden se cancela, orders-service llama a POST /api/inventory/:sku/adjust?delta=+N para devolver las unidades al stock.

Stored procedures que usa

fn_adjust_stock(sku, delta) — ajuste atómico con validación de stock negativo  |  fn_inventory_report() — clasificación por nivel de stock

:8084 shipping-service — El trazador de envíos

Convierte una orden confirmada en un envío físico con número de seguimiento único. Es el punto de contacto entre el sistema digital y la operación logística real.

Por qué es irremplazable

• Genera el número de tracking TRACK-XXXXXXXX (basado en UUID truncado) que es la identidad del envío en el mundo físico, imprimible en etiquetas y legible en la vista del transportista.

• Implementa la validación de doble factor en la entrega: para marcar un envío como ENTREGADO, el transportista debe proveer el customerCode (SL-XXXXXX) que el cliente le muestra, Y el recipientRut del receptor. Ambos se verifican cruzando contra orders.client_code y customers.rut respectivamente. Esto previene entregas fraudulentas.

• Genera el código QR del envío en formato PNG (SMARTLOGIX-{tracking_number}) que el transportista puede escanear y el cliente puede verificar al recibir.

• Notifica automáticamente al notification-service en cada cambio de etapa — y lo hace de forma resiliente: si la notificación falla, el envío igual se crea (ver test de resiliencia).

• Llama al orders-service para cruzar el código de cliente al confirmar entrega — comunicación síncrona HTTP entre microservicios.

:8085 notification-service — El registro de trazabilidad

Registra cada evento significativo del ciclo de vida de un pedido. Es el sistema de auditoría distribuida que permite a cualquier actor (cliente, operador, transportista) ver el historial completo de un envío.

Por qué es irremplazable

• Implementa idempotencia mediante una restricción UNIQUE en (event_id, target_audience): si el mismo evento se intenta registrar dos veces (retry de red, duplicado de mensajes), el sistema devuelve 409 DUPLICATE sin crear registros espurios — crítico en sistemas distribuidos donde las retransmisiones son normales.

• Maneja múltiples audiencias: customer (alias CLIENT), ops, shipper, system, y la audiencia BOTH por defecto cuando no se especifica — permitiendo filtrar notificaciones por tipo de usuario.

• Provee la línea de tiempo completa de un pedido vía GET /api/notifications/order/:id ordenada cronológicamente — la vista que el cliente ve en el portal de tracking y que operaciones usa para auditoría.

• Es el receptor pasivo del patrón Saga: no inicia flujos, solo registra. Esto lo hace altamente cohesivo y fácil de escalar o sustituir sin afectar la lógica de negocio.

Innovación: integración con API externa

Sistema de notificaciones por email — Nodemailer + SMTP

SmartLogix integra un sistema de email transaccional mediante Nodemailer, que actúa como cliente SMTP hacia un proveedor de correo externo (Gmail, Outlook, SendGrid). Esta integración convierte al sistema logístico en una plataforma de comunicación activa con el cliente — no solo registra datos, notifica en tiempo real.

Cuándo se dispara

Al crear una orden

Cuando orders-service genera el código SL-XXXXXX y persiste la orden en la base de datos, llama inmediatamente al módulo shared/email.js para enviar un correo al cliente con:

• Su código de cliente único (SL-XXXXXX)
• El SKU y cantidad pedida
• Instrucciones para rastrear su pedido

Arquitectura del módulo

shared/email.js

Módulo compartido en la capa Backend/shared/. Configura un transporter SMTP con variables de entorno (EMAIL_USER, EMAIL_PASS, EMAIL_HOST) y expone la función sendOrderConfirmation().

Los secretos NUNCA están hardcodeados — se leen en runtime desde el entorno Docker.

Por qué es innovador

Comunica el SL-XXXXXX

El código de cliente es el eje de seguridad del sistema: permite rastrear el pedido Y valida la entrega. Sin un mecanismo confiable de comunicarlo al cliente, el sistema de doble factor de entrega no funciona.

El email resuelve ese problema: el código llega al cliente en el mismo momento en que la orden se crea.

Resiliencia

Fallo no rompe el flujo

El envío de email es asíncrono y no bloquea la respuesta al frontend. Si el servidor SMTP está caído, la orden igual se crea y el cliente recibe su 201 — el email puede reintentarse o el operador puede comunicar el código manualmente.

// Backend/shared/email.js — módulo de email transaccional const nodemailer = require('nodemailer'); const transporter = nodemailer.createTransport({ host: process.env.EMAIL_HOST, // smtp.gmail.com / smtp.sendgrid.net port: 587, secure: false, auth: { user: process.env.EMAIL_USER, // leído desde entorno Docker pass: process.env.EMAIL_PASS, // nunca hardcodeado } }); async function sendOrderConfirmation(email, customerCode, sku, quantity) { await transporter.sendMail({ from: `"SmartLogix" <${process.env.EMAIL_USER}>`, to: email, subject: `Tu pedido fue creado — Código: ${customerCode}`, html: buildEmailTemplate(customerCode, sku, quantity), }); }

Generación de códigos QR — librería qrcode

El shipping-service genera un código QR por cada envío usando el paquete qrcode de npm. El contenido del QR es SMARTLOGIX-{tracking_number}. El endpoint GET /api/shipments/:id/qr devuelve la imagen en base64 PNG que el frontend renderiza directamente en la pantalla del transportista para ser escaneada al entregar el paquete.

Por qué es relevante

• El QR es el vínculo entre el mundo físico (etiqueta impresa en el paquete) y el sistema digital.

• Permite confirmar que el paquete físico corresponde al envío digital antes de entregarlo — capa adicional de verificación de identidad del pedido.

• Es generado on-demand, no almacenado — menor superficie de ataque y siempre actualizado.

Impacto de la innovación: La combinación email (SL-XXXXXX al crear) + QR (SMARTLOGIX-TRACK al despachar) + doble factor (SL + RUT al entregar) forma una cadena de verificación completa que cubre todo el ciclo logístico con integración de APIs externas reales en cada etapa crítica.

Flujo completo de un pedido (paso a paso)

1

Ops crea cliente — POST /api/customers

orders-service persiste nombre, email, teléfono, dirección y RUT en orders_db.customers. El RUT es necesario para la validación de entrega al final del flujo.

2

Ops agrega producto — POST /api/inventory

inventory-service registra el SKU con stock inicial en inventory_db.inventory.

3

Ops crea orden — POST /api/orders

orders-service genera el código SL-XXXXXX (6 caracteres aleatorios base36 en mayúsculas, prefijados con "SL-"), persiste la orden en estado CREATED y dispara el envío de email al cliente con su código único. Responde con 201 { orderId, customerCode, status: "CREATED" }.

4

Ops confirma orden — PUT /api/orders/:id/confirm (Saga orquestado)

Este es el paso más complejo. orders-service actúa como orquestador y ejecuta dos llamadas HTTP en secuencia:

Paso 4a: POST inventory-service/api/inventory/:sku/adjust?delta=-N — descuenta stock.

Paso 4b: POST shipping-service/api/shipments — crea el envío con tracking TRACK-*.

Paso 4c: orders-service actualiza el estado a EN_PREPARACION.

Si algún paso falla, el response incluye warnings pero la orden avanza — compensación manual o reintento posterior.

5

Ops asigna transportista — PUT /api/orders/:id/assign?transporter=X

Se graba el campo assigned_to en la orden. La vista del transportista en el frontend filtra exactamente las órdenes donde assignedTo === session.username.

6

Transportista sale a repartir — PUT /api/shipments/:id/stage?stage=EN_REPARTO

shipping-service cambia la etapa y notifica al notification-service. El evento queda registrado en la línea de tiempo del pedido.

7

Transportista confirma entrega — PUT /api/shipments/:id/stage?stage=ENTREGADO

Body requerido: { customerCode: "SL-XXXXXX", recipientRut: "12.345.678-9" }. shipping-service valida dos factores independientes: cruza el customerCode contra orders.client_code (consultando orders-service) y el recipientRut contra customers.rut. Solo si ambos coinciden cambia el estado a ENTREGADO.

8

Cliente rastrea su pedido — GET /api/orders/track/:clientCode

Endpoint público (sin auth). Devuelve estado, SKU, fecha y nombre del cliente — sin email ni teléfono. El cliente conoce su SL-XXXXXX porque lo recibió por email al crear la orden.

El patrón de mock es idéntico en los 4 servicios: jest.mock('../shared/db') → mockQuery = jest.fn() → mockQuery.mockResolvedValueOnce({'{'}rows:[...]{'}'}). Esto garantiza tests deterministas, rápidos (ms, no segundos) y sin dependencia de infraestructura.

Registrar evento — POST /api/notifications

Qué: Persistencia de eventos con idempotencia y manejo de audiencias Crítico: la idempotencia previene registros duplicados en sistemas distribuidos
T1 — Persiste notificación → 201 SUCCESS: El mock simula la inserción exitosa. El test verifica que el response incluye status: "SUCCESS" y los datos del evento. Es el test del camino feliz.

T2 — Audiencia default = BOTH cuando no se especifica: Si el campo audience no viene en el body, el servicio lo asigna como BOTH. Este comportamiento por defecto es fundamental: un evento como ORDER_CREATED debe notificar tanto al cliente como a operaciones. Sin este test, un cambio descuidado podría dejar eventos sin audiencia.

T3 — Alias customer → CLIENT: El sistema acepta audience: "customer" pero lo almacena como CLIENT en la DB (el nombre canónico del enum). El test verifica la normalización — sin esto, las queries por audiencia fallarían al no encontrar el valor "customer".

T4 — Idempotencia: mismo eventId + audience → 409 DUPLICATE:
// Primer intento → SUCCESS mockQuery.mockResolvedValueOnce({ rows: [saved] }); const r1 = await request(app).post('/api/notifications').send(validNotification); expect(r1.body.status).toBe('SUCCESS'); // Segundo intento con mismo eventId → DUPLICATE mockQuery.mockResolvedValueOnce({ rows: [] }); // SELECT encuentra duplicado const r2 = await request(app).post('/api/notifications').send(validNotification); expect(r2.status).toBe(409); expect(r2.body.status).toBe('DUPLICATE');
En sistemas distribuidos, una misma petición puede llegar dos veces (retry de red, timeout con reintento). Sin idempotencia, la línea de tiempo de un pedido podría tener eventos duplicados confundiendo al cliente.

T5 — Idempotencia por constraint DB (code 23505) → 409 DUPLICATE:
mockQuery .mockResolvedValueOnce({ rows: [] }) // SELECT no encuentra duplicado .mockRejectedValueOnce(Object.assign(new Error('unique violation'), { code: '23505' })); // La INSERT falla por constraint UNIQUE en la DB expect(res.status).toBe(409); expect(res.body.status).toBe('DUPLICATE');
Este test cubre el caso de race condition: dos peticiones concurrentes pasan la verificación SELECT pero la segunda falla en INSERT por la restricción UNIQUE de PostgreSQL. El código de error 23505 es el código estándar de PostgreSQL para violación de unicidad — el servicio lo intercepta y devuelve 409 en lugar de 500.

T6 — Audiencia UNKNOWN → 400: { audience: "UNKNOWN" } rechazado. Solo audiencias del enum son válidas — evita registros con audiencia inválida que nunca serían consultados.

T7 — Incluye message en el registro: El campo message se persiste correctamente. El frontend lo muestra en la línea de tiempo del pedido.

T8 — Health check UP/DEGRADED: Mismo patrón. Si notification-service está DEGRADED, otros servicios pueden tomar decisiones (reintentar, no bloquear, alertar).
